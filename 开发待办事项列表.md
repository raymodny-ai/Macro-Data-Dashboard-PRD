# 宏观流动性与资产定价状态识别系统 - 开发待办事项列表（重构版）

## 项目概述

本项目旨在构建"状态识别者"宏观监控终端，通过自动化采集五组硬核物理数据（通胀二阶导、财政增量、流动性走廊、AI CapEx、市场传染），实现独立双轨控制模型大屏的实时可视化。系统将帮助机构投资者在清算主义范式下精准识别市场状态，区分"健康疼痛区"与"功能瘫痪区"。

**核心技术栈**: React/Vue + Three.js/ECharts + FastAPI/Python + Airflow/Celery + TimescaleDB/Redis

### 五大架构原则

1. **数据管道与抗脆弱调度层 (ETL & Airflow)**：T-180 滚动全量覆盖 + 禁用 Catchup + 双层限流隔离，确保数据状态与官方修正后的绝对一致。
2. **存储引擎与计算下推层 (TimescaleDB)**：UPSERT 无锁覆写 + 按需刷新连续聚合 + Python 重度计算分离，兼顾海量写入与复杂跨资产关联查询。
3. **业务流转与后端通信层 (FastAPI & Redis)**：SSE 主动推送替代轮询 + 事件驱动缓存强驱逐 + 独立规则引擎热更新，极端行情下毫秒级状态推送。
4. **前端大屏与 3D 渲染优化层 (React/Vue + Three.js)**：压缩二维数组传输 + 前端本地 BufferGeometry + WebGL 显式 GC + LOD 降级，7×24 挂机零内存泄漏。
5. **物理架构与本地算力底座**：全栈 Docker Compose 本地部署 + OpenClaw 内网穿透，充分利用本地 DDR5 内存与计算资源，物理隔离保障数据安全。

## 附录

### A. 关键技术决策记录

1. **为何选择 TimescaleDB 而非 ClickHouse**：本系统需要复杂的关系型 JOIN 查询（如 SEC 元数据关联流动性利差），TimescaleDB 基于 PostgreSQL 原生支持 ACID 事务和标准 SQL，配合 UPSERT 机制实现无锁覆写，而 ClickHouse 在联表查询和事务合规性上存在短板。

2. **为何采用 Airflow + Celery 双层调度与双层限流隔离**：Airflow 负责宏观层面的 DAG 编排（T-180 全量拉取任务），宽容接口（FRED/Fiscal Data）直接在 Worker 中处理；Celery 负责微观层面的 SEC EDGAR 严格限流（≤10次/秒），两者互补，确保不互相干扰。

3. **为何前端选用 Three.js + ECharts 组合**：Three.js 擅长 3D 曲面和自定义 WebGL 渲染，ECharts 擅长 2D 统计图表和交互，两者结合可覆盖所有可视化需求。

4. **为何采用 T-180 滚动全量覆盖而非增量更新**：幂等性优先于效率。FRED 等官方数据源会对历史数据进行修正（Revision），增量更新无法捕捉修正，而 T-180 全量覆盖可确保数据状态与官方绝对一致，算力换取数据可靠性。

5. **为何废弃前端轮询改用 SSE**：轮询方案在极端行情下延迟过高（每 5 分钟一次），且无法保证服务器主动推送警报的实时性。SSE 在保持轻量级的同时，支持服务器单向主动推送，满足毫秒级状态广播需求。

6. **为何选择本地部署 + OpenClaw 内网穿透而非云端部署**：本系统高频的 Pandas 内存计算和 T-180 全量数据吞吐对内存带宽要求极高，云端数据库的存储与流出带宽（Egress）溢价成本远超本地硬件投资。本地部署同时实现物理隔离的数据安全保障。

7. **为何 3D 数据在后端压缩、前端解压而非直接传输网格**：完整的 BufferGeometry 网格数据（vertices/faces/colors）体积庞大，网络传输开销高。改为传输压缩二维数组，由浏览器本地生成几何体，大幅降低带宽占用，同时充分利用客户端 GPU 算力。
